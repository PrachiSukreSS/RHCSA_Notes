## 2️⃣ **File Permissions Basics**

### Understanding `ls -l` Output
```bash
[root@rhel ~]# ls -l /usr/bin/passwd
-rwsr-xr-x. 1 root root 27856 Jan 30 2020 /usr/bin/passwd
```

**Permission Breakdown:**
- `-` = regular file
- `rws` = owner permissions (read, write, **setuid**)
- `r-x` = group permissions
- `r-x` = others permissions

### Standard Permissions (octal)
| Permission | Symbol | Octal |
|------------|--------|-------|
| Read | r | 4 |
| Write | w | 2 |
| Execute | x | 1 |

## 3️⃣ **Special Permissions**

### **SETUID (s)** - 4000 or u+s
When set on executable files, runs with owner's privileges.

```bash
# Remove SETUID from passwd
[root@rhel ~]# chmod u-s /usr/bin/passwd
[root@rhel ~]# ls -l /usr/bin/passwd
-rwxr-xr-x. 1 root root 27856 Jan 30 2020 /usr/bin/passwd
```

**What happens?** Without SETUID, regular users CANNOT change their passwords!

### **SETGID (s)** - 2000 or g+s
- On **files**: Runs with group owner's privileges
- On **directories**: New files inherit directory's group

```bash
# Apply SGID to directory
[root@rhel ~]# chmod g+s /data/impdata
[root@rhel ~]# ls -ld /data/impdata
drwxrws--x. 2 amol redhat 6 Mar 31 11:20 /data/impdata
#           ^ s in group position indicates SGID
```

### **Sticky Bit (t)** - 1000 or o+t
Users can only delete files they own within the directory.

```bash
# Apply Sticky Bit
[root@rhel ~]# chmod o+t /data/impdata
[root@rhel ~]# ls -ld /data/impdata
drwxrws--t. 2 amol redhat 6 Mar 31 11:20 /data/impdata
#           ^ t in others position indicates Sticky Bit
```

## 4️⃣ **Hands-on Practical: /data/impdata Exercise**

### Step-by-Step Commands from Your Notes:

```bash
# 23. Change ownership to amol
[root@rhel ~]# chown amol /data/impdata

# 24. Verify ownership
[root@rhel ~]# ls -ld /data/impdata
drwxr-xr-x. 2 amol root 6 Mar 31 11:15 /data/impdata

# 25. Change group to redhat
[root@rhel ~]# chgrp redhat /data/impdata

# 26. Verify group change
[root@rhel ~]# ls -ld /data/impdata
drwxr-xr-x. 2 amol redhat 6 Mar 31 11:15 /data/impdata

# 27. Set permissions to 771 (rwxrwx--x)
[root@rhel ~]# chmod 771 /data/impdata/
# Owner: rwx (7), Group: rwx (7), Others: --x (1)

# 28. Verify permissions
[root@rhel ~]# ls -ld /data/impdata
drwxrwx--x. 2 amol redhat 6 Mar 31 11:15 /data/impdata

# 29. Switch to user amol
[root@rhel ~]# su - amol

# 30. Check permissions from amol's perspective
[amol@rhel ~]$ ls -ld /data/impdata/
drwxrwx--x. 2 amol redhat 6 Mar 31 11:15 /data/impdata/

# 31. Apply SGID (from amol or root)
[amol@rhel ~]$ chmod g+s /data/impdata

# 32. Verify SGID is set
[amol@rhel ~]$ ls -ld /data/impdata/
drwxrws--x. 2 amol redhat 6 Mar 31 11:15 /data/impdata/
#           ^ s in group position

# 33. Switch to red1
[amol@rhel ~]$ su - red1

# 34. Check directory (should have group access)
[red1@rhel ~]$ ls -ll /data/impdata

# 35. Switch back to amol
[red1@rhel ~]$ su - amol

# 36. Apply Sticky Bit
[amol@rhel ~]$ chmod o+t /data/impdata/

# 37. Verify Sticky Bit
[amol@rhel ~]$ ls -ld /data/impdata/
drwxrws--t. 2 amol redhat 6 Mar 31 11:15 /data/impdata/
#           ^ t in others position

# 38. Switch to red1 again
[amol@rhel ~]$ su - red1

# 39. View command history
[red1@rhel ~]$ history
```

## 5️⃣ **Creating and Testing Special Permissions**

### Lab: /mydata Directory

```bash
# Create new directory
[root@rhel ~]# mkdir /mydata

# Apply SETUID (doesn't affect directories)
[root@rhel ~]# chmod u+s /mydata
# Result shows no effect on directories

# Apply SETGID
[root@rhel ~]# chmod g+s /mydata

# Apply Sticky Bit
[root@rhel ~]# chmod o+t /mydata

# Check final permissions
[root@rhel ~]# ls -ld /mydata
drw-r--r--. 2 root root 6 Mar 31 11:30 /mydata
# Wait - this doesn't match! Let's fix:

# Better way - set all at once:
[root@rhel ~]# chmod 3777 /mydata
[root@rhel ~]# ls -ld /mydata
drwxrwsr-t. 2 root root 6 Mar 31 11:30 /mydata
# 3 = SETGID(2) + Sticky(1) = 3
# 777 = full permissions for all
```

## 6️⃣ **Password Management Issues (Troubleshooting)**

### Common Password Errors from Your Notes:

```bash
# Error: "Authentication token manipulation error"
[amit@telnet ~]$ password amit
Current password:
new password:
mypassword
Authentication token manipulation error
Password: password unchanged
```

**Causes & Solutions:**

1. **Password too short/weak**
   ```bash
   # Check password policy
   [root@telnet ~]# cat /etc/security/pwquality.conf
   ```

2. **SETUID missing from /usr/bin/passwd**
   ```bash
   # Fix: Re-add SETUID
   [root@telnet ~]# chmod u+s /usr/bin/passwd
   [root@telnet ~]# ls -l /usr/bin/passwd
   -rwsr-xr-x. 1 root root 27856 Jan 30 2020 /usr/bin/passwd
   ```

3. **Wrong password format**
   ```bash
   # Correct way to change password
   [amit@telnet ~]$ passwd
   Changing password for user amit.
   Current password:
   New password:
   Retype new password:
   passwd: all authentication tokens updated successfully.
   ```

## 7️⃣ **Permission Summary Table**

| Command | Effect | Example |
|---------|--------|---------|
| `chmod 755 file` | rwxr-xr-x | Standard file |
| `chmod u+s file` | Add SETUID | Run as owner |
| `chmod g+s dir` | Add SGID | Inherit group |
| `chmod o+t dir` | Add Sticky Bit | Delete own files only |
| `chown user:group file` | Change owner:group | `chown amol:redhat file` |
| `chgrp group file` | Change group only | `chgrp redhat file` |

```bash
# Create test directory with sticky bit

# Now only file owners can delete their files
```

### Exercise 3: Fix Password Change Issue

```bash
# Verify passwd has SETUID

# If missing SETUID, fix it
chmod u+s /usr/bin/passwd
```

## 📝 **Key Takeaways**

1. **SETUID** (`u+s`) - Runs executables with file owner's permissions
2. **SETGID** (`g+s`) - On directories: new files inherit group ownership
3. **Sticky Bit** (`o+t`) - Prevents users from deleting others' files
4. **Always verify** permissions with `ls -l` after changes
5. **Password issues** usually mean missing SETUID on `/usr/bin/passwd`
